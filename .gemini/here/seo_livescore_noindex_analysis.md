# 라이브스코어(리그·팀·선수·매치) SEO Indexability 및 DB 구조 분석 보고서

현재 서비스의 라이브스코어 데이터 규모(매치 3.6만 건, 선수 2.9만 건, 팀 6천 건 이상)를 고려할 때, 모든 페이지를 무조건 구글에 인덱싱(색인)하는 것은 **색인 비대화(Index Bloat)**와 **크롤링 예산(Crawl Budget) 낭비**를 초래하여 사이트 전체 SEO에 부정적인 영향을 줄 수 있습니다.

본 보고서에서는 현재 구현된 index/noindex 판정 시스템과 이를 제어하는 DB 구조를 분석하고, 최적화 방안을 제안합니다.

## 1. 현재 데이터베이스 상태 및 분석 결과

스크립트([explore_seo_db.js](file:///home/kim/.gemini/antigravity-cli/brain/33e812a3-338a-41d8-8ffd-b506221aa6a5/scratch/explore_seo_db.js))를 실행하여 수집한 실제 데이터 현황은 다음과 같습니다.

### 1) DB 테이블 규모
* **전체 리그 (Leagues)**: 38개
* **전체 팀 (Football Teams)**: 6,448개
* **전체 선수 (Football Players)**: 29,856명
* **전체 매치 (Fixtures)**: 36,432경기

### 2) 현재 색인 제어 현황 (`leagues.is_major` 기준)
* **색인 허용 리그 (`is_major = true`)**: **37개** (월드컵, EPL, 라리가, 분데스리가, K리그 1·2, J1 리그, 사우디 리그, 덴마크 리그 등 대부분 포함)
* **색인 제외 리그 (`is_major = false`)**: **1개** (`클럽 친선경기` 단 1개만 제외 상태)
* **예외 색인 팀 테이블 (`team_indexable_competitions`)**: 총 **4,324개** 레코드 등록됨 (주요 컵 대회 참가 팀 등)

> [!WARNING]
> **핵심 문제점**: 현재 설정으로는 총 38개 리그 중 37개 리그가 "주요 리그(Major)"로 설정되어 있어, 거의 모든 매치(3.6만), 선수(2.9만), 팀(6천) 페이지가 구글 인덱싱 대상이 됩니다. 이는 총 **70,000개 이상의 유동적인 URL**을 의미하며, 신규/중소 규모 사이트가 감당하기에는 크롤링 예산 낭비 및 품질 저하 위험이 매우 큽니다.

## 2. 코드 및 판정 로직 분석

[`seoIndexability.ts`](file:///home/kim/web2/src/domains/livescore/actions/seoIndexability.ts)에 구현된 판정 로직은 매우 유연하고 훌륭하게 설계되어 있습니다.

### 1) 리그 및 매치 색인 판정 (`resolveFootballIndexability`)
* **쿼리 파라미터가 있을 때**: 정렬, 필터 등으로 인한 중복 URL은 `noindex` 처리 (`query-state`)
* **비정상 경기**: 취소(`CANC`), 연기(`PST`), 중단(`SUSP`) 경기 등은 자동 `noindex` 처리 (`match-status`)
* **일반 경기**: 리그의 `is_major` 컬럼이 `true`인 경우에만 색인 허용 (`non-indexable-league` 차단)

### 2) 팀 및 선수 색인 판정 (`resolveTeamIndexability`)
* **예외 처리**: 해당 팀이 `team_indexable_competitions` 테이블에 존재한다면(예: UCL 참가팀 등) 소속 리그의 중요도와 무관하게 색인 허용
* **기본 처리**: 그렇지 않다면 소속된 기본 리그의 `is_major` 상태를 따라감

## 3. SEO 최적화 제안 (Index Bloat 방지)

구글봇은 고유한 텍스트 콘텐츠가 적고 실시간 스코어 수치만 있는 데이터형 페이지를 대량으로 수집할 때 "색인 생성됨(텍스트 없음)" 혹은 "크롤링됨 - 현재 색인 생성되지 않음" 상태로 분류하며 사이트 품질 점수를 낮게 평가할 수 있습니다. 

따라서 **국내 사용자가 검색창에 직접 검색할 확률이 높은 "핵심 1티어 리그"만 색인을 허용**하고, 나머지는 `noindex, follow` 처리하여 크롤링 효율성을 집중해야 합니다.

### 제안: 리그별 `is_major` 데이터 플래그 다이어트

DB의 `leagues` 테이블에서 아래와 같이 중요도가 다소 떨어지는 리그들의 `is_major` 값을 `false`로 일괄 업데이트할 것을 강력히 권장합니다.

| 분류 | 리그 목록 (is_major = true 유지) | 권장 조치 |
| :--- | :--- | :--- |
| **티어 1 (유지)** | 월드컵, EPL, 라리가, 분데스리가, 세리에A, 리그앙, K리그1, K리그2, 챔피언스리그, 유로파리그, FA컵 | `is_major = true` 유지 |
| **티어 2 (검토)** | J1 리그, 사우디 프로리그, MLS, 사우디 리그 등 | 서비스 타겟 및 트래픽 전략에 따라 결정 |
| **티어 3 (해제)** | 덴마크 수페르리가, 중국 슈퍼리그, 리가 MX, 에레디비시, 프리메이라 리가, 챔피언십, 스코틀랜드 프리미어십, 기타 군소 컵대회 등 | `is_major = false`로 변경 권장 |

### 기대 효과
1. **서치콘솔 색인 수집율 개선**: 저품질 데이터 페이지가 제거되면서, 핵심 분석 글과 커뮤니티 페이지의 색인 반영 속도가 월등히 빨라집니다.
2. **크롤링 최적화**: 구글봇이 무의미한 경기 기록 대신 실제 트래픽이 나오는 경기와 선수 페이지 위주로 크롤링을 수행합니다.
3. **손쉬운 관리**: 코드를 수정하지 않고 **DB에서 특정 리그의 `is_major` 컬럼 값을 `false`로 바꾸는 것만으로** 해당 리그 소속 매치, 팀, 선수 페이지 수천 개가 자동으로 `noindex` 전환됩니다. (설계 구조가 아주 훌륭합니다.)
